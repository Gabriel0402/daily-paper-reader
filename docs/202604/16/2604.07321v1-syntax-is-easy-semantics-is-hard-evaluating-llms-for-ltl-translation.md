## 摘要
命题线性时序逻辑（Propositional Linear Temporal Logic，LTL）是一种广泛用于为软件、网络和系统指定期望需求及安全与隐私策略的形式化方法。然而，由于其复杂的语义结构，将此类需求和策略用 LTL 表达仍然具有一定挑战性。由于许多安全与隐私分析工具要求以 LTL 公式作为输入，这种难度使得许多开发者和分析人员难以使用这些工具。大型语言模型（LLMs）通过将自然语言片段翻译为 LTL 公式，可能有助于扩大这类工具的可及性。本文通过评估若干具有代表性的 LLM，将陈述性英语句子翻译为 LTL 公式的有效性，以验证这一假设。我们使用人工生成和合成的真实数据，从语法与语义两个维度评估其表现。结果揭示了三点发现：（1）与先前研究一致，LLM 在 LTL 的语法层面表现优于语义层面；（2）更为详细的提示通常能提升模型表现；（3）将该任务重构为 Python 代码补全问题可以显著提高整体性能。我们还讨论了在该任务中实现公平评估所面临的挑战，并在最后提出了对未来研究的建议。

## Abstract
Propositional Linear Temporal Logic (LTL) is a popular formalism for specifying desirable requirements and security and privacy policies for software, networks, and systems. Yet expressing such requirements and policies in LTL remains challenging because of its intricate semantics. Since many security and privacy analysis tools require LTL formulas as input, this difficulty places them out of reach for many developers and analysts. Large Language Models (LLMs) could broaden access to such tools by translating natural language fragments into LTL formulas. This paper evaluates that premise by assessing how effectively several representative LLMs translate assertive English sentences into LTL formulas. Using both human-generated and synthetic ground-truth data, we evaluate effectiveness along syntactic and semantic dimensions. The results reveal three findings: (1) in line with prior findings, LLMs perform better on syntactic aspects of LTL than on semantic ones; (2) they generally benefit from more detailed prompts; and (3) reformulating the task as a Python code-completion problem substantially improves overall performance. We also discuss challenges in conducting a fair evaluation on this task and conclude with recommendations for future work.

## 二、方法论与技术路线

- **总体思路**：  
  构建一个系统性的评估框架，比较不同 LLM 在“NL→LTL”任务的语法与语义准确性；同时区分三种交互接口（prompt interface），分析提示复杂度与任务结构对性能的影响。

- **关键技术设计**：
  1. **六层评价框架**：针对语法和语义，分别构造若干子任务，确保结果可从多维度量化。
  2. **三种接口设计**：
     - **Minimal Interface**：最简提示，仅提供任务说明；
     - **Detailed Interface**：包含正式语法（BNF Grammar）、操作符语义定义与示例；
     - **Python (AST) Interface**：将 LTL 公式表达为 Python 抽象语法树（AST），在模型输出层面强制结构化。
  3. **自动化语义验证**：  
     使用 *NuSMV* 模型检测器作为“语义 oracle”，验证生成的 LTL 是否与真值公式等价或满足逻辑蕴含关系。  
     即检查 𝜑_pred ⇔ 𝜑_gt 及单向蕴含 𝜑_pred ⇒ 𝜑_gt 或 𝜑_gt ⇒ 𝜑_pred。
  4. **任务定义**：
     - AP 提取（atomic propositions extraction）；
     - LTL 式子构造（generation）；
     - LTL 式子合法性判定（well-formedness）；
     - 轨迹判定与生成（trace classification & generation）；
     - Past-LTL 构造（含过去时操作符）。

## 六、主要结论与发现

- **主要发现**：
  1. **LLM 在语法层面表现良好，语义层面显著不足。**  
     平均语法正确率高达 70–90%，但语义等价率通常仅约 20–30%。
  2. **提示复杂度显著影响性能。**  
     Detailed Prompt 比 Minimal 提升约 10–20%，Python/AST 表达再次提升至 60%以上。
  3. **Python 代码补全形式显著改善结果。**  
     将 NL→LTL 视作代码生成任务时，语法与语义等价率均提升显著（最高达 72%）。
  4. **时序操作符误用为主要错误来源。**  
     模型常混淆 G/F/X 等运算符的作用范围，导致语义错位。
  5. **安全领域任务更困难。**  
     在 VERIFY 子集上，语义等价度下降约 3–16 个百分点，AP 映射缺失时下降更大。

- **整体结论**：  
  当前 LLM 可以生成语法正确的 LTL，但在语义层面仍不可靠；其输出需经符号验证器（如 NuSMV）校验。将任务设计为结构化代码生成可增强稳健性，但无法完全弥合语义差距。
