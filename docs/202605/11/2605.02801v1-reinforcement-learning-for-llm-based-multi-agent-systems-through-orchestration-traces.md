## 摘要
随着大语言模型（LLM）智能体从孤立的工具使用者演变为协同团队，强化学习（RL）不仅必须优化单个动作，还必须优化工作的生成、委派、通信、聚合和停止方式。本文通过编排轨迹（orchestration traces）研究基于 LLM 的多智能体系统强化学习：编排轨迹是时间交互图，其事件包括子智能体生成、委派、通信、工具使用、返回、聚合和停止决策。通过这一视角，我们确定了三个技术轴。首先，奖励设计涵盖八个系列，包括针对并行加速、拆分正确性和聚合质量的编排奖励。其次，奖励和信用信号附着在从 token 到团队的八个信用或信号承载单元上；在我们整理的资源池中，显式的反事实消息级信用分配仍然特别稀缺。第三，编排学习分解为五个子决策：何时生成、委派给谁、如何通信、如何聚合以及何时停止。在我们截至 2026 年 5 月 4 日的整理池中，我们没有发现针对停止决策的显式强化学习训练方法。我们将学术方法与来自 Kimi Agent Swarm、OpenAI Codex 和 Anthropic Claude Code 的公开工业证据联系起来。由此产生的规模差距是公开报告的部署范围与开放学术评估体系之间的差距，而非对工业训练轨迹的独立验证。我们在 https://github.com/xxzcc/awesome-llm-mas-rl 发布了研究成果，包括一个包含 84 篇带标签论文的资源池、一个 32 条记录的排除日志、脚本化的语料库统计数据，以及一个用于可重放编排轨迹的最小 JSON 模式。

## Abstract
As large language model (LLM) agents evolve from isolated tool users into coordinated teams, reinforcement learning (RL) must optimize not only individual actions but also how work is spawned, delegated, communicated, aggregated, and stopped. This paper studies RL for LLM-based multi-agent systems through orchestration traces: temporal interaction graphs whose events include sub-agent spawning, delegation, communication, tool use, return, aggregation, and stopping decisions.   Using this lens, we identify three technical axes. First, reward design spans eight families, including orchestration rewards for parallelism speedup, split correctness, and aggregation quality. Second, reward and credit signals attach to eight credit- or signal-bearing units from token to team; explicit counterfactual message-level credit remains especially sparse in our curated pool. Third, orchestration learning decomposes into five sub-decisions: when to spawn, whom to delegate to, how to communicate, how to aggregate, and when to stop. In our curated pool as of May 4, 2026, we found no explicit RL training method for the stopping decision.   We connect academic methods to public industrial evidence from Kimi Agent Swarm, OpenAI Codex, and Anthropic Claude Code. The resulting scale gap is a gap between publicly reported deployment envelopes and open academic evaluation regimes, not independent verification of industrial training traces. We release the artifact at https://github.com/xxzcc/awesome-llm-mas-rl, including an 84-entry tagged paper pool, a 32-record exclusion log, scripted corpus statistics, and a minimal JSON schema for replayable orchestration traces.

*   **三大技术轴：**
    1.  **奖励设计（8大系列）：** 从共享团队奖励（R1）到具体的编排奖励（R7，如并行加速比、完成率），并强调混合奖励（R8）的权重退火策略。
    2.  **信用分配（8个层级）：** 建立了从“团队”到“Token”的层级结构，重点讨论了“编排者级”、“角色级”和“消息级”的信用分配。
    3.  **编排学习（5类决策）：** 明确了编排者需学习的五个子决策：何时生成（O1）、委派给谁（O2）、如何通信（O3）、如何聚合（O4）以及何时停止（O5）。

### 3. 实验设计
本论文主要是一篇**综述与分类学研究**，而非单一算法的实验论文。其“实验”体现在对现有文献和工业系统的深度审计：
*   **数据集/场景：** 涵盖了代码生成（SWE-Bench）、网页浏览（WebArena）、科学研究、数学推理等多个领域。
*   **Benchmark 分析：** 审计了现有的多智能体基准（如 MultiAgentBench, TAMAS），指出它们大多只关注任务成功率（E1），而缺乏对并行效率（E2）、协作质量（E3）和协议开销（E4）的评估。
*   **对比对象：** 将学术界的方法（如 M-GRPO, Dr.MAS, SHARP）与工业界系统（Kimi K2.5/2.6, OpenAI Codex, Claude Code）进行对比，重点分析了部署规模和轨迹长度。

### 4. 资源与算力
由于本文是综述性质的论文，**未明确说明具体的训练算力消耗**（如 GPU 型号或训练时长）。
*   但是，论文在第 5 节专门讨论了**推理/采样成本（Rollout Cost）**，给出了成本估算公式，并指出多智能体 RL 的主要瓶颈在于采样阶段的墙钟时间，这通常比单智能体推理高出 1-2 个数量级。

### 5. 实验数量与充分性
*   **样本量：** 论文构建了一个包含 **84 篇** 核心论文的资源池（从 116 篇候选记录中筛选），并记录了 32 条排除日志。
*   **充分性：** 调研覆盖了从 2024 年 Q4 到 2026 年 Q2 的最新进展，时间跨度约 18 个月。
*   **客观性：** 论文通过脚本化的元分析（Scripted Meta-analysis）生成统计数据，并公开了带有 18 个维度标签的 CSV 数据库，具有较高的透明度和可审计性。

### 6. 论文的主要结论与发现
1.  **停止决策的空白：** 在所有调研的方法中，没有发现任何显式使用 RL 训练“何时停止（O5）”决策的方法，通常依赖硬性步数限制。
2.  **规模差距（Scale Gap）：** 工业界（如 Kimi）已报告支持 300 个子智能体、4000 步协作的系统，而学术界方法大多停留在 10-100 步的规模。
3.  **信用扩散问题：** 在长轨迹中，简单的共享奖励会导致信用信号极其微弱（SNR 低），必须引入角色级或消息级的反事实信用分配。
4.  **编排奖励的瞬态性：** 工业界证据显示，并行加速等编排奖励应作为训练初期的“脚手架”，在后期应退火至零，以避免智能体为了刷分而进行伪并行。

### 7. 优点
*   **系统性强：** 首次将 LLM-MAS RL 抽象为“编排轨迹”，为该领域提供了统一的术语表和分类框架。
*   **工业与学术桥接：** 不仅分析论文，还深入拆解了 Kimi、Codex 等实际产品的技术文档，具有极强的实战参考价值。
*   **开源贡献：** 提供了机器可读的 JSON Schema 和自动验证工具，方便后续研究者标准化记录多智能体交互。

### 8. 不足与局限
*   **主观性风险：** 论文由独立研究员完成，分类标签（如“信用粒度”）主要基于个人阅读理解，缺乏多标注者的一致性校验。
*   **缺乏原生实验：** 论文未提出并验证一个新的算法，所有的结论均基于对他人工作的二次分析。
*   **时间偏差：** 论文标注日期为 2026 年 5 月（相对于当前时间点属于未来预测或特定设定），其引用的部分“未来”文献可能在当前真实时空中尚未完全发表或经过同行评审。
