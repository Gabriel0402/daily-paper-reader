## 摘要
我们提出了 TraceFix，这是一个针对大语言模型（LLM）多智能体协作的验证优先流水线。智能体首先根据任务描述将协议拓扑合成结构化中间表示（IR），生成 PlusCal 协作逻辑，并利用 TLA+ 模型检查器（TLC）提供的反例迭代修复协议，直到验证通过。验证后的过程体被编译为各智能体的系统提示词，并在运行时监控器下执行，该监控器会拒绝拓扑结构之外的协作操作。在涵盖 16 个场景系列的 48 个任务中，所有任务均通过了完整的 TLC 验证；其中 62.5% 的任务在首次尝试时即通过，且没有任务需要超过四次修复迭代。尽管状态空间跨越六个数量级，但每个任务的验证均在 60 秒内完成。一项包含 3,456 次运行的运行时对比显示，受拓扑监控的执行实现了最高的任务完成率（平均 89.4%，完全完成 81.5%）；且当模型能力降低时，使用验证协议的运行时间退化率仅为纯提示词和纯对话基线的约一半。在固定运行时的配对消融实验表明，经 TLC 验证的协议将死锁/活锁（DL/LL）比例从 31.1% 降低至 14.1%，且在故障注入情况下表现出最大的性能差距。

## Abstract
We present TraceFix, a verification-first pipeline for Large Language Model (LLM) multi-agent coordination. An agent synthesizes a protocol topology as a structured intermediate representation (IR) from a task description, generates PlusCal coordination logic, and iteratively repairs the protocol using counterexamples from the TLA+ model checker (TLC) until verification succeeds. Verified process bodies are compiled into per-agent system prompts and executed under a runtime monitor that rejects out-of-topology coordination operations. On 48 tasks spanning 16 scenario families, all tasks reach full TLC verification; 62.5% pass on the first attempt and none requires more than four repair iterations. State spaces span six orders of magnitude yet verification completes in under 60 s for every task. A 3,456-run runtime comparison shows that topology-monitored execution achieves the highest task completion (89.4% average, 81.5% full) and that runtimes using the verified protocol degrade at roughly half the rate of prompt-only and chat-only baselines when model capability is reduced. A paired ablation under a fixed runtime shows that TLC-verified protocols cut deadlock/livelock (DL/LL) from 31.1% to 14.1%, with the largest separation under fault injection.

#### 1. 核心问题与整体含义
在多智能体系统（MAS）中，大语言模型（LLM）智能体通常需要并发运行、异步通信并共享外部状态。研究发现，系统的主要失败原因往往不在于单个智能体的能力，而在于**协作协议的缺陷**。由于并发执行的交错（Interleaving）情况极其复杂，智能体容易陷入死锁（Deadlock）、竞态条件（Race Conditions）或过早终止等故障。这些 Bug 具有隐蔽性，仅在特定的调度序列下触发。TraceFix 的动机在于引入形式化验证（Formal Verification）的严谨性，通过“验证优先”的思路，在协议执行前识别并修复这些潜在的并发风险。

#### 2. 方法论
TraceFix 提出了一个四阶段的验证与修复流水线：
*   **拓扑合成（Topology Synthesis）**：智能体首先将任务描述转化为结构化的中间表示（IR），定义参与者（Agents）、共享资源（Locks/Counters）和有向通信信道（Channels）。
*   **逻辑生成（Logic Generation）**：基于拓扑 IR，生成 PlusCal（一种类算法的形式化语言）协作逻辑。PlusCal 能够显式表达非确定性分支（either/or），使模型检查器能探索所有可能的决策路径。
*   **模型检查与验证（Model Checking）**：将 PlusCal 转换为 TLA+ 规范，利用 TLC 检查器在有限边界内穷举所有状态。检查属性包括：死锁自由、互斥访问、无孤立锁（Orphan Locks）以及终止时信道清空。
*   **反例驱动修复（Counterexample-driven Repair）**：若验证失败，TLC 会生成一个最小反例轨迹（Trace）。修复智能体分析该轨迹，定位协议中的逻辑漏洞（如循环等待），并迭代修改 PlusCal 代码，直到通过验证。
*   **运行时监控（Runtime Monitoring）**：验证后的协议编译为智能体提示词。在执行阶段，一个轻量级监控器会拦截并拒绝任何违反预定义拓扑结构的协作操作（如访问未声明资源）。

#### 3. 实验设计
*   **Benchmark 设计**：构建了包含 16 个场景系列（如软件开发、医疗咨询、半导体制造、CI/CD 流水线等）的基准测试，每个场景分为易、中、难三个等级，共 48 个任务。
*   **对比架构**：
    1.  **Topology-monitored (TraceFix)**：验证协议 + 拓扑监控。
    2.  **Mediator-enforced**：通过中心化协调器强制执行，智能体无自主权。
    3.  **Prompt-only**：仅提供协议提示词，无运行时监控。
    4.  **Chat-only**：智能体仅通过自由聊天协作，无显式协议工具。
*   **模型与变量**：使用 Claude Opus 4.6 进行协议设计；运行时测试使用 `gpt-5-mini` 和 `gpt-5-nano`；引入了不同等级的故障注入（Fault Injection）来测试鲁棒性。

#### 5. 实验数量与充分性
*   **实验规模**：共进行了 **3,456 次** 运行时实验（48 任务 × 4 架构 × 3 故障等级 × 3 重复 × 2 模型）。
*   **充分性**：实验覆盖了从简单锁竞争到复杂异步信道握手的多种协作模式。通过配对消融实验（Paired Ablation）隔离了“验证修复”这一变量，证明了反例反馈的直接贡献。

#### 6. 主要结论与发现
*   **修复收敛性**：100% 的任务最终通过了 TLC 验证。62.5% 首次通过，其余在 4 次迭代内全部修复成功。
*   **性能提升**：TraceFix 架构实现了最高的任务完成率（平均 89.4%），显著优于 Chat-only（75.0%）。
*   **安全性**：验证后的协议将死锁/活锁率从 31.1% 降低至 14.1%。
*   **能力缓冲（Capability Buffer）**：当使用较弱模型（nano 级别）时，拥有验证协议的系统性能退化速度仅为无协议系统的一半，说明形式化协议能补偿模型推理能力的不足。

#### 7. 优点
*   **闭环保证**：将形式化验证的“硬约束”与 LLM 生成的“灵活性”结合，通过反例提供了精准的调试信号。
*   **可扩展性**：拓扑 IR 与 PlusCal 的分离使得该方法可以适配不同的 LLM 和底层执行环境。
*   **鲁棒性**：在故障注入环境下表现优异，验证协议为智能体提供了结构化的错误恢复路径。

#### 8. 不足与局限
*   **语义与安全脱节**：验证仅保证协作安全性（不锁死），不保证任务语义的正确性（如智能体可能安全地达成了一个错误的结果）。
*   **边界限制**：验证是在有界假设下进行的（如信道深度限制），超出边界可能仍存在未发现的 Bug。
*   **语义漂移（Semantic Drift）**：在处理超时导致的修复时，智能体可能会为了通过验证而简化协议逻辑，从而偏离原始任务的复杂需求。
*   **监控粒度**：目前的运行时监控仅在拓扑级别（信道/资源是否存在），尚未实现对步骤顺序（Label-level）的完全强制约束。
