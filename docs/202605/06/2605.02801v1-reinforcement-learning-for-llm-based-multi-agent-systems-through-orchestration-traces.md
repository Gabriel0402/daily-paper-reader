## 摘要
随着大语言模型（LLM）智能体从孤立的工具使用者演变为协同团队，强化学习（RL）不仅必须优化单个动作，还必须优化任务的生成、委派、通信、聚合和停止方式。本文通过编排轨迹（orchestration traces）研究基于 LLM 的多智能体系统强化学习：编排轨迹是时间交互图，其事件包括子智能体生成、委派、通信、工具使用、返回、聚合和停止决策。通过这一视角，我们确定了三个技术轴。首先，奖励设计涵盖八个系列，包括针对并行加速、拆分正确性和聚合质量的编排奖励。其次，奖励和信用信号附着在从 token 到团队的八个信用或信号承载单元上；在我们整理的文献库中，显式的反事实消息级信用分配仍然特别稀缺。第三，编排学习分解为五个子决策：何时生成、委派给谁、如何通信、如何聚合以及何时停止。在我们截至 2026 年 5 月 4 日整理的文献库中，尚未发现针对停止决策的显式强化学习训练方法。我们将学术方法与来自 Kimi Agent Swarm、OpenAI Codex 和 Anthropic Claude Code 的公开工业证据联系起来。由此产生的规模差距反映了公开报告的部署范围与开放学术评估体系之间的鸿沟，而非对工业训练轨迹的独立验证。我们在 https://github.com/xxzcc/awesome-llm-mas-rl 发布了相关成果，包括一个包含 84 篇带标签论文的资源库、一个 32 条记录的排除日志、脚本化的语料库统计数据，以及一个用于可重放编排轨迹的最小 JSON 模式。

## Abstract
As large language model (LLM) agents evolve from isolated tool users into coordinated teams, reinforcement learning (RL) must optimize not only individual actions but also how work is spawned, delegated, communicated, aggregated, and stopped. This paper studies RL for LLM-based multi-agent systems through orchestration traces: temporal interaction graphs whose events include sub-agent spawning, delegation, communication, tool use, return, aggregation, and stopping decisions.   Using this lens, we identify three technical axes. First, reward design spans eight families, including orchestration rewards for parallelism speedup, split correctness, and aggregation quality. Second, reward and credit signals attach to eight credit- or signal-bearing units from token to team; explicit counterfactual message-level credit remains especially sparse in our curated pool. Third, orchestration learning decomposes into five sub-decisions: when to spawn, whom to delegate to, how to communicate, how to aggregate, and when to stop. In our curated pool as of May 4, 2026, we found no explicit RL training method for the stopping decision.   We connect academic methods to public industrial evidence from Kimi Agent Swarm, OpenAI Codex, and Anthropic Claude Code. The resulting scale gap is a gap between publicly reported deployment envelopes and open academic evaluation regimes, not independent verification of industrial training traces. We release the artifact at https://github.com/xxzcc/awesome-llm-mas-rl, including an 84-entry tagged paper pool, a 32-record exclusion log, scripted corpus statistics, and a minimal JSON schema for replayable orchestration traces.

    1.  **奖励设计（8 大族群）：** 从共享团队奖励（R1）到具体的编排奖励（R7，如并行加速比、完成率），并强调了混合奖励（R8）在实际中的重要性。
    2.  **信用分配（8 个层级）：** 构建了从“团队 -> 编排者 -> 角色 -> 智能体 -> 回合 -> 消息 -> 工具 -> Token”的信用层级。指出目前“消息级”和“编排者级”的信用分配研究最为稀缺。
    3.  **编排学习（5 个子决策）：** 明确了编排者需要学习的五个核心决策（O1-O5）：何时生成、委派给谁、如何通信、如何聚合、何时停止。
*   **算法流程：** 总结了三种训练范式：(A) 冻结子智能体仅训练编排者；(B) 联合训练但使用智能体归一化优势；(C) 使用中心化评论家（Central Critic）的完全解耦训练。

#### 3. 实验设计
本论文是一篇**综述与分类学研究**，其“实验”主要体现为对现有文献和工业系统的系统性审计与对比：

*   **数据集/场景：** 涵盖了代码生成（SWE-Bench）、数学推理、网络浏览（WebArena）、工具使用（ToolBench）等多个领域。
*   **文献池（Paper Pool）：** 筛选了 116 篇相关记录，最终保留并标记了 84 篇核心条目（包括 51 篇焦点论文和 33 篇基础文献）。
*   **工业锚点：** 对比了 Moonshot AI 的 **Kimi Agent Swarm**（K2.5/2.6）、OpenAI **Codex** 和 Anthropic **Claude Code** 的公开技术细节。
*   **对比维度：** 评估了学术方法在轨迹长度（T）和团队规模（K）上与工业界报告数据的差距。

#### 4. 资源与算力
由于本文为综述性质，**未直接进行模型训练**，因此没有报告作者本人的 GPU 使用情况。
*   但论文深入分析了 **Rollout 成本**：指出多智能体 Rollout 的成本远超单智能体，受限于最慢的子智能体和复杂的依赖图。
*   文中引用了 Kimi K2.5 的数据，指出其支持多达 100 个子智能体和 1500 个协调步骤，这暗示了工业级训练需要极高的算力支持和工程优化（如流水线并行）。

#### 5. 实验数量与充分性
*   **文献审计：** 论文对 84 篇论文进行了 18 个维度的详细标记，这种横向对比非常充分，涵盖了从 2024 年 Q4 到 2026 年 Q2 的最新进展。
*   **客观性：** 论文诚实地指出了学术界与工业界之间的“规模鸿沟”——学术论文通常在 T < 100 的短轨迹上实验，而工业界已触及 T > 1000 的场景。
*   **局限性：** 实验分析主要基于公开论文和企业技术博客，由于工业界（如 OpenAI）不公开具体训练细节，部分分析属于基于部署形态的推断。

#### 6. 主要结论与发现
*   **规模差距：** 学术评估体系与工业部署环境在轨迹长度和并行规模上存在数量级差距。
*   **研究空白：** 在编排决策中，**“何时停止”（O5）** 的显式 RL 训练在现有文献中几乎缺失；**“消息级反事实信用分配”** 极度稀缺。
*   **奖励设计：** 编排奖励（R7）通常作为训练早期的“脚手架”，在后期需退火至零，以避免智能体为了刷分而过度生成。
*   **安全风险：** 多智能体系统引入了新的攻击面，如跨智能体的提示词注入污染和共享内存中毒。

#### 7. 优点
*   **理论框架清晰：** 提出的“编排轨迹”和“信用层级”为混乱的多智能体研究提供了统一的语言。
*   **工业结合紧密：** 不仅仅停留在学术论文，还深入分析了 Kimi、Codex 等实际产品的工程约束。
*   **资源开放：** 发布了带标签的论文库、JSON 模式和验证脚本，具有很高的工具价值。
*   **前瞻性：** 提出了 15 个明确的开放研究问题，为后续研究指明了方向。

#### 8. 不足与局限
*   **缺乏原生实验：** 论文未提出并验证一个新的算法，更多是归纳总结。
*   **标注偏差：** 论文标签由作者手工标注，虽然有审计日志，但仍存在主观判断风险。
*   **工业数据不透明：** 依赖于企业的技术博客和宣传材料，无法独立验证工业级训练轨迹的真实细节。
*   **基准测试缺失：** 论文指出目前缺乏能同时测量“任务成功率、并行效率、协作质量、协议开销”的 MAS 原生基准测试。
