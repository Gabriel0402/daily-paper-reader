## 摘要
随着语言模型越来越多地被部署为自主代理，它们必须在开放式的交互任务中与其他人进行长期协调。然而，现有的评估很少同时测试这些需求，而是侧重于单智能体任务、短期交互或高度结构化的多智能体场景。我们提出了 $alem$，一个基于JAX的开放式多智能体协调基准测试，构建在类似Craftax的动态上。Alem将程序生成的协调任务、软专业化、通信和可控制的协调难度嵌入到一个具有探索、制作、交易和战斗的长期生存世界中。我们在同质团队中零样本评估了 $13$ 个现代LLM，以训练过的MARL代理作为参考点。当前的LLM代理距离解决Alem还很远，平均标准化回报仅为约6%，但它们的失败并非均匀分布。在最难的协调设置中，零样本的Gemini-3.1-Pro-High接近训练了十亿步的MARL代理，而GPT-5.4-High在基础任务奖励上得分很高，但协调奖励却低得多。这种对比表明，个体任务能力并不等同于协调能力。消融实验显示，通信是对协调贡献最大的因素，而记忆和推理在用于维持多步计划时有所帮助。总体而言，我们的结果将协调确定为前沿LLM代理的一个独特瓶颈，与单智能体能力分离。Alem使这个瓶颈变得可测量，并为开发能够通信、分配角色和执行共享计划的代理提供了一个受控的测试平台。代码可在 https://github.com/alem-world/alem-env 获取。

## Abstract
As language models are increasingly deployed as autonomous agents, they must coordinate with others over long horizons in open-ended interactive tasks. Yet existing evaluations rarely test these demands together, instead emphasising single-agent tasks, short interactions, or highly structured multi-agent settings. We introduce $alem$, a JAX-based benchmark for open-ended multi-agent coordination built on Craftax-like dynamics. Alem embeds procedurally generated coordination tasks, soft specialisation, communication, and controllable coordination difficulty into a long-horizon survival world with exploration, crafting, trading, and combat. We evaluate $13$ modern LLMs zero-shot within homogeneous teams, with trained MARL agents as reference points. Current LLM agents remain far from solving alem, averaging only ~6% normalised return, but their failures are not uniform. On the hardest coordination setting, zero-shot Gemini-3.1-Pro-High approaches MARL agents trained for one billion steps, while GPT-5.4-High achieves strong base-task reward but much lower coordination reward. This contrast shows that individual task competence does not imply coordination competence. Ablations show that communication is the largest contributor to coordination, while memory and reasoning help when used to maintain multi-step plans. Overall, our results identify coordination as a distinct bottleneck for frontier LLM agents, separate from single-agent capabilities. Alem makes this bottleneck measurable and provides a controlled testbed for developing agents that communicate, allocate roles, and execute shared plans. Code is available at https://github.com/alem-world/alem-env.

### 核心思想
- 基于Craftax-like动力学，用JAX实现一个开放式多智能体世界ALEM（阿姆哈拉语中“世界”的意思），将协调任务从隐式变为显式、程序化生成，且难度可控。

### 关键技术细节
- **环境建模**：部分可观察随机博弈（POSG），包含多个代理、状态、动作、观测、转移和奖励。
- **协调类型**：覆盖从长期依赖（如资源链）、短窗口交接（handover）到同步联合行动（synchronous）的完整谱系。
- **程序化协调生成**：每个episode独立采样协调任务（采矿、建造、战斗、制作），分配协调类型（同步/交接）和所需代理数量（2到N），防止记忆化。
- **软专业化（Soft Specialisation）**：任何代理均可尝试任何角色动作，但专家成功率100%，非专家成功率为1-α（α为难度参数）。鼓励角色分配同时允许灵活适应。
- **难度控制**：单一标量α∈[0,1]控制执行难度（如N-代理概率、交接窗口宽度、非专家效率等），α=0.3/0.6/0.9分别对应Easy/Medium/Hard，不改变世界布局和协调机会频率，从而隔离难度变化。
- **通信信道**：代理可广播离散消息，用于发送意图、分配任务、请求资源等。
- **接口**：提供像素、符号和文本三种接口。文本接口用于LLM评估，包含系统提示、历史观察、当前状态、动作空间、推理/消息/笔记字段。
- **奖励**：密集健康奖励 + 稀疏成就奖励（包括27个新协调成就）。协调奖励仅授予参与代理。

## 6. 论文的主要结论与发现

1. **ALEM远未解决**：LLM平均Total%仅约6%，且不同模型间差异大（Llama-3.1-8B仅0.8%，Gemini-3.1-Pro-High达18.1% Easy）。
2. **基础能力不保证协调能力**：GPT-5.4-High基础奖励高（11.8% Hard），但协调奖励仅4.2%；而Gemini-3.1-Pro-High协调奖励17.5%接近MARL 1B的17.6%。个体任务胜任不等于协调胜任。
3. **通信是协调最大贡献因素**：移除通信使Gemini协调奖励从17.5%降至5.3%，Gemma从8.8%降至3.8%。
4. **记忆和推理有帮助但依赖使用方式**：Gemini用scratchpad做前向计划时帮助大；Gemma主要做状态摘要，效果不明显。推理降低会损害两者的基础和协调性能。
5. **异质团队表现接近成员平均水平**：混合团队既不能达到最强成员水平，也不会拖累到最弱成员，但存在额外的协调摩擦。
6. **大多数LLM未充分利用难度轴**：总回报在不同难度下几乎持平，说明它们无法有效利用协调机制来应对更高难度。

## 7. 优点：方法或实验设计上的亮点

- **程序化生成协调任务**：每次episode独立采样，避免记忆化和模式利用，更接近真实开放式协调。
- **单一可控难度参数**：α改变执行约束而不改变世界布局和协调机会，允许严格隔离协调难度效应。
- **软专业化设计**：不硬性限制角色，非专家仍可尝试但成功率低，促进动态角色分配。
- **完整协调谱系**：覆盖从长期依赖到同步行动，比现有基准（如SMAC、Overcooked、Melting Pot）更全面。
- **JAX实现高效**：支持大规模MARL训练（10亿步单GPU两天内），便于重现和扩展。
- **多接口支持**：像素、符号、文本三种观察通道，同时兼容MARL和LLM评估。
- **统计报告严谨**：使用bootstrap置信区间，报告均值和不确定性。
- **诊断分析丰富**：包括协调类型分解、成就层级覆盖、生存时长与回报关系、会话风格分析等，深入揭示失败模式。

## 8. 不足与局限

- **模型覆盖不完整**：未评估视觉语言模型（VLM）和Claude系列；闭源模型（GPT、Gemini）仅10种子，统计上可能不够稳健。
- **异质团队实验初步**：只有两种混合组合（同家族和跨家族），且未探索更多异构团队规模和配置。
- **缺乏跨episode记忆和终身学习**：评估仅在单episode内进行，未测试长期记忆和适应性。
- **部分消融不完全**：Gemini的推理消融仅能降低而不能完全关闭，效果解释受到限制。
- **文本接口依赖**：零样本LLM评估依赖固定提示模板，可能未充分优化模型性能（但保持了公平性）。
- **环境复杂度与真实场景差异**：ALEM是模拟生存游戏，虽然具备开放式特性，但与真实世界多智能体交互（如机器人协作、社会模拟）仍有距离。
